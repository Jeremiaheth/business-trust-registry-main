# Moderation Boundary

## Scope

BTR-NG public moderation stays narrow. The public beta supports fact correction and public profile stewardship, not broad complaint handling.

## Allowed Public Workflows

- claim a public business profile
- correct non-sensitive public metadata
- point to public links and hashes
- request review when a profile is based on thin or outdated public evidence

## Rejected Public Workflows

- evidence file uploads
- personal data submissions
- private contact exchanges
- third-party conduct complaints
- requests for public adjudication outside fact correction

## Operator Triage

1. Confirm the submission stays within the public lane.
2. Reject anything that includes personal data, attachments, or confidential material.
3. Accept only the minimum public-safe references needed for review.
4. Record the disposition in the repo-friendly workflow already defined by issue templates and dispute records.

## Public Language Rules

- Describe the system as an evidence dossier, not an authority.
- Keep public notes based on available verified evidence.
- Make uncertainty visible when queue pressure, stale procurement data, or under-review status affects scoring.

## Escalation Rule

If a submission cannot be handled safely as a public fact-correction case, move it out of the public lane rather than stretching repo scope.
